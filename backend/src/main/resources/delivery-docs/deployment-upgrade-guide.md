# 部署与升级手册

本文档面向律师事务所电子档案系统的实施与运维人员，适用于镜像交付场景。

## 1. 交付边界

- 交付物以 Docker 镜像、Docker Compose 清单、环境变量文件和本手册为准。
- 客户现场不要求依赖源码仓库阅读部署说明。
- 业务管理员负责系统内配置、档案、借阅和备份恢复。
- 程序升级、容器重启、数据库迁移和回滚由服务器运维管理员执行。

## 2. 推荐目录规范

- 源码目录：`/root/src/law-firm-archive`
- 部署目录：`/opt/law-firm-archive`

要求：

- `root/src` 只存放源码、构建脚本和版本标签信息。
- `/opt/law-firm-archive` 只存放部署清单、环境变量和持久化数据。

## 3. 推荐部署方式

推荐采用 `Docker Compose + 私有镜像仓库`。

原因：

- 适合当前单机和轻量化容器环境
- 升级、回滚路径清晰
- 便于面向多个律所客户重复交付

## 4. 首次部署

1. 创建部署目录 `/opt/law-firm-archive`
2. 准备 `docker-compose.registry.yml`
3. 准备 `docker/.env.registry`
4. 确认 `APP_VERSION`、`APP_COMMIT_SHA`、`APP_BUILD_TIME`
5. 执行 `docker compose pull`
6. 执行 `docker compose up -d`
7. 完成部署后执行冒烟测试

## 5. 升级流程

1. 升级前先备份数据库、对象存储和关键配置
2. 确认目标版本号，例如 `v0.1.7`
3. 在部署目录更新镜像标签和版本元数据
4. 执行 `docker compose pull`
5. 执行 `docker compose up -d`
6. 执行部署后冒烟测试
7. 记录测试结果和是否回滚

## 6. 回滚流程

1. 确认上一稳定版本标签
2. 将部署清单切回上一稳定标签
3. 重新执行 `docker compose up -d`
4. 如有必要恢复备份数据
5. 重新执行冒烟测试并记录结果

## 7. 版本号要求

建议统一使用可追踪标签：

- `v0.1.0`
- `v0.1.1`
- `v0.1.7`

不建议使用：

- `latest`
- `test`
- `new`

## 8. 发布留痕要求

每次首次部署、升级、回滚至少记录：

- 客户名称
- 环境名称
- 部署版本
- 镜像标签
- Git 提交号
- 执行时间
- 执行人
- 升级前备份编号
- 冒烟测试结果
- 是否回滚
