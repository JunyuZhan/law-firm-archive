## 一、律所业务概述

### 1.1 律所是什么？

律师事务所是律师执业的机构，类似于会计师事务所、设计院。律所的核心业务是为客户提供法律服务。

**律所的主要收入来源**：
- 诉讼代理费（打官司）
- 非诉法律服务费（尽职调查、合同审核、并购等）
- 常年法律顾问费
- 专项法律服务费

### 1.2 律所的组织结构

```
律师事务所
├── 合伙人（Partner）          # 律所的所有者/股东，参与利润分配
│   ├── 管理合伙人             # 负责律所日常管理
│   └── 权益合伙人             # 享有分红权的合伙人
├── 律师（Lawyer）
│   ├── 主办律师               # 案件的主要负责人（项目角色）
│   └── 协办律师               # 协助主办律师工作（项目角色）
├── 实习律师（Trainee）        # 尚未取得执业证的法学毕业生
├── 行政人员
│   ├── 财务                   # 负责收款、开票、提成计算
│   ├── 行政                   # 负责后勤、档案管理
│   └── 前台/秘书              # 接待、文秘
└── 律师助理（Paralegal）      # 辅助律师工作，非律师身份
```

### 1.3 两个独立的维度（重要概念）

#### 1.3.1 维度一：项目角色（Project Role）

| 角色 | 英文 | 说明 | 职责 |
|------|------|------|------|
| **主办律师** | Lead Lawyer | 案件的第一负责人 | 对案件质量负责，统筹全局，决定团队分配 |
| **协办律师** | Associate | 协助主办律师 | 执行具体任务，协助办理案件 |
| **案源人** | Originator | 带来案源的人 | 介绍客户/案件，可以是团队成员也可以不是 |

**项目角色的规则**：
- 每个项目必须有且仅有一个主办律师
- 协办律师可以有多个（0-N个）
- 案源人可以是任何人（律师或非律师）
- 主办律师可以同时是案源人
- 协办律师也可以同时是案源人

#### 1.3.2 维度二：薪酬模式（Compensation Type）

| 类型 | 英文 | 收入构成 | 典型人员 |
|------|------|----------|----------|
| **提成律师** | Commission | 案件提成为主，无/低底薪 | 合伙人、资深律师 |
| **授薪律师** | Salaried | 固定月薪，不参与提成 | 新律师、部分律师 |
| **混合制** | Hybrid | 底薪 + 提成 | 过渡期律师 |

**薪酬模式的规则**：
- 薪酬模式由律所（admin/director）设定，律师本人不能修改
- 薪酬模式决定是否参与项目提成分配
- 与项目角色无关：授薪律师也可以做主办律师

#### 1.3.3 组合示例

```
【示例1】提成律师做主办
张律师（提成制）担任主办律师
→ 张律师获得办案提成中的主办份额

【示例2】授薪律师做主办
李律师（授薪制）担任主办律师
→ 李律师拿固定月薪，不获得提成
→ 办案提成由其他提成制的协办律师分配

【示例3】提成律师做协办
王律师（提成制）担任协办律师
→ 王律师获得办案提成中的协办份额

【示例4】授薪律师做协办
赵律师（授薪制）担任协办律师
→ 赵律师拿固定月薪，不获得提成
→ 其份额留给其他提成制律师或计入律所

【示例5】混合团队
项目团队：
- 主办律师：张律师（提成制）
- 协办律师A：李律师（提成制）
- 协办律师B：王律师（授薪制）
- 案源人：陈律师（提成制）

提成分配：
- 律所留存：30%
- 案源提成：陈律师获得
- 办案提成：张律师 + 李律师分配（王律师不参与）
```

#### 1.3.4 薪酬模式对提成分配的影响

| 场景 | 提成律师 | 授薪律师 |
|------|----------|----------|
| 作为主办律师 | 获得主办份额的提成 | 不获得提成（拿月薪） |
| 作为协办律师 | 获得协办份额的提成 | 不获得提成（拿月薪） |
| 作为案源人 | 获得案源提成 | 获得案源提成（这点可配置）|
| 分配比例设置 | 正常设置 | 设为0%（系统建议） |

> **特殊说明**：授薪律师作为案源人是否获得案源提成，可由律所自行规定。有些律所允许，有些不允许。系统应支持配置。

```java
// 项目角色枚举（与薪酬无关）
public enum TeamRole {
    LEAD_LAWYER("LEAD_LAWYER", "主办律师"),
    ASSOCIATE("ASSOCIATE", "协办律师"),
    ORIGINATOR("ORIGINATOR", "案源人"),
    PARALEGAL("PARALEGAL", "律师助理");
}

// 薪酬模式枚举（与项目角色无关）
public enum CompensationType {
    COMMISSION("COMMISSION", "提成制", "按案件收入提成"),
    SALARIED("SALARIED", "授薪制", "固定月薪"),
    HYBRID("HYBRID", "混合制", "底薪+提成");
}
```

### 1.3 律所业务的核心特点

| 特点 | 说明 | 对系统的影响 |
|------|------|-------------|
| **项目制** | 每个案件/项目独立核算 | 需要项目维度的数据统计 |
| **团队协作** | 一个案件多人参与 | 需要团队成员管理和工时记录 |
| **提成分配** | 收入需要按比例分配 | 复杂的提成计算逻辑 |
| **文档密集** | 大量法律文书和证据 | 强大的文档管理能力 |
| **期限敏感** | 诉讼有严格的法定期限 | 期限提醒功能非常重要 |
| **保密要求** | 客户信息严格保密 | 数据权限和访问控制 |
| **利益冲突** | 不能代理对立双方 | 利冲审查是必要流程 |

## 二、核心业务术语

### 2.1 人员相关术语

| 术语 | 英文 | 说明 |
|------|------|------|
| **合伙人** | Partner | 律所的所有者，承担无限责任，参与利润分配 |
| **主办律师** | Lead Lawyer | 案件的第一负责人，对案件质量负责 |
| **协办律师** | Associate | 协助主办律师处理案件的律师 |
| **实习律师** | Trainee | 持实习证，尚未正式执业的律师 |
| **案源人** | Originator | 带来案件的人，可能是律师或其他人 |
| **执业证号** | License No. | 律师执业资格证书编号，全国唯一 |

### 2.2 项目/案件相关术语

| 术语 | 英文 | 说明 |
|------|------|------|
| **案件/项目** | Matter/Case | 律所承接的法律服务项目，是核心业务单元 |
| **收案** | Case Intake | 正式承接案件，立案登记 |
| **结案** | Case Close | 案件服务完成，进入归档流程 |
| **案号** | Case Number | 法院立案后分配的编号，如(2024)京0105民初1234号 |
| **案由** | Cause of Action | 案件的类型，如"合同纠纷"、"劳动争议" |
| **标的额** | Amount in Dispute | 诉讼争议的金额 |
| **管辖法院** | Court | 有权审理案件的法院 |

### 2.3 诉讼相关术语

| 术语 | 说明 |
|------|------|
| **原告** | 提起诉讼的一方 |
| **被告** | 被起诉的一方 |
| **第三人** | 与案件有利害关系的其他当事人 |
| **一审** | 第一次审判 |
| **二审/上诉** | 对一审判决不服，向上级法院上诉 |
| **再审** | 对已生效判决申请重新审理 |
| **执行** | 判决生效后，强制执行阶段 |
| **举证期限** | 法院规定的提交证据截止日期 |
| **开庭** | 法院开庭审理案件 |
| **判决书** | 法院作出的判决文书 |
| **诉讼时效** | 提起诉讼的有效期限（一般3年） |

### 2.4 财务相关术语

| 术语 | 说明 |
|------|------|
| **代理费/律师费** | 客户支付的法律服务费用 |
| **风险代理** | 根据案件结果收费，胜诉才收费或收取更高比例 |
| **计时收费** | 按律师工作小时数收费 |
| **固定收费** | 约定固定金额收费 |
| **提成** | 律师从收入中分得的报酬 |
| **案源提成** | 给案源人的提成 |
| **办案提成** | 给办案律师的提成 |
| **到账** | 客户付款到律所账户 |
| **核销** | 将到账款项与具体项目/客户关联 |

### 2.5 文档相关术语

| 术语 | 说明 |
|------|------|
| **起诉状** | 原告向法院提交的起诉文书 |
| **答辩状** | 被告对起诉状的书面答复 |
| **代理词** | 律师在庭审中的代理意见 |
| **法律意见书** | 律师出具的专业法律意见 |
| **委托书** | 授权律师代理的法律文书 |
| **证据** | 证明案件事实的材料 |
| **质证** | 对对方证据发表意见 |

#### 5.1.3 利益冲突审查规则

**什么是利益冲突？**
律师不能同时代理利益对立的双方。例如：
- 不能代理原告打被告，又代理被告打原告
- 不能为A公司做尽调，同时代理B公司收购A公司

**利冲审查规则**：
```
触发条件：新建项目时必须进行利冲审查

检索范围：
1. 检索新项目的客户是否曾经是对方当事人
2. 检索新项目的对方当事人是否曾经是我方客户
3. 检索关联企业（股东、子公司等）

审查结果：
- 无冲突 → 可以收案
- 有冲突 → 需要说明情况，申请豁免或拒绝收案
- 潜在冲突 → 需要进一步核实
```

**利冲审查流程**：

**风险代理规则**（重要）：
```
风险代理是指案件胜诉后才收费或收取更高比例

限制条件：
1. 刑事案件禁止风险代理
2. 行政案件禁止风险代理
3. 婚姻家庭案件禁止风险代理
4. 风险代理比例一般不超过30%

计算示例：
- 基础费：10,000元（无论输赢都收）
- 风险费：胜诉金额的10%
- 若胜诉获得100万，风险费 = 100万 × 10% = 10万
- 总收费 = 10,000 + 100,000 = 110,000元
```

```java
        // 这些比例在 MatterCommissionConfig 中，主办律师无权修改
        
        // 3. 验证：只能调整办案人员的内部分配比例
        // 4. 验证：办案人员分配比例之和必须等于100%
        // 5. 验证：授薪律师的分配比例必须为0
        BigDecimal total = BigDecimal.ZERO;
        for (TeamDistributionCommand dist : distributions) {
            User user = userRepository.findById(dist.getUserId()).orElseThrow();
            if ("SALARIED".equals(user.getCompensationType()) && 
                dist.getDistributionRate().compareTo(BigDecimal.ZERO) > 0) {
                throw new BusinessException("授薪律师不参与提成分配，比例必须为0");
            }
            total = total.add(dist.getDistributionRate());
        }
        if (total.compareTo(new BigDecimal("100")) != 0) {
            throw new BusinessException("团队分配比例之和必须等于100%");
        }
        
        // 6. 保存调整记录
        // ...
    }
}
```

#### 5.5.3 质证说明

**什么是质证？**
质证是指诉讼当事人对对方提交的证据发表意见，说明是否认可该证据。

**质证三性**：
```
1. 真实性：证据本身是否真实
   - 认可：承认证据是真实的
   - 否认：认为证据是伪造的
   - 不清楚：无法确认真假

2. 合法性：证据取得方式是否合法
   - 如：偷录的录音可能被认定为非法证据

3. 关联性：证据是否与案件事实有关
   - 即使证据真实、合法，如果与案件无关也不采纳
```

## 七、常见业务问题FAQ

### 7.1 客户相关

**Q1: 一个人可以既是客户又是对方当事人吗？**
> A: 可以。同一个人/企业在不同案件中可能是不同角色。例如A公司在案件1中是我方客户，在案件2中可能是对方当事人。这就是为什么需要利冲审查。

**Q2: 客户和联系人是什么关系？**
> A: 客户是服务对象（个人或企业），联系人是客户方具体对接的人。例如客户是某公司，联系人可能是该公司的法务、财务等。

### 7.2 项目相关

**Q3: 案件和项目有什么区别？**
> A: 在系统中统一用"项目"（Matter）表示，诉讼类项目也可以叫"案件"。主要区别：
> - 诉讼类项目有案号、对方当事人、法院等信息
> - 非诉类项目没有这些信息

**Q4: 为什么一个项目需要多个阶段？**
> A: 诉讼案件会经历多个法定程序阶段，每个阶段的工作内容和期限不同。例如一审阶段有举证期限，二审阶段有上诉期限。

**Q5: 主办律师和案源人有什么区别？**
> A: 
> - 主办律师：负责案件具体办理的律师
> - 案源人：介绍/引荐客户的人（可能是律师、也可能是其他人）
> - 两者可以是同一人，也可以不同

### 7.3 财务相关

**Q6: 为什么需要核销？直接记录收款不行吗？**
> A: 律所会有多个项目、多个客户，一笔到账需要明确是哪个项目的款项，才能：
> 1. 计算该项目的应收账款
> 2. 触发对应项目的提成计算
> 3. 统计项目的实际收入

**Q7: 提成比例是固定的吗？**
> A: 不是。提成比例由律所制定，可能根据以下因素变化：
> - 业务类型（诉讼/非诉）
> - 项目金额大小
> - 律师级别
> - 特殊约定

### 7.4 文档与证据

**Q8: 文档和证据有什么区别？**
> A: 
> - 文档：广义的法律文书，包括合同、代理词、工作记录等
> - 证据：用于证明案件事实的材料，需要在法庭上出示
> - 关系：证据是特殊类型的文档，有更严格的管理要求（编号、质证等）

**Q9: 为什么证据需要记录"三性"意见？**
> A: 法庭质证时，对每份证据都要从真实性、合法性、关联性三个角度发表意见，这是法定程序要求。系统需要记录双方的质证意见，便于后续查阅。

### 7.5 档案相关

**Q10: 为什么案件结案后还要保存档案这么多年？**
> A: 
> 1. 法律规定：律师法和档案法有明确的保管期限要求
> 2. 业务需要：客户可能会后续咨询、再审等
> 3. 风险防范：应对可能的执业投诉、诉讼

## 八、开发注意事项

### 8.1 数据权限控制要点

```
数据权限级别：
1. 全部数据 - 合伙人、财务主管可见
2. 部门数据 - 部门负责人可见本部门数据
3. 团队数据 - 只能看自己参与的项目
4. 个人数据 - 只能看自己创建/负责的数据

特殊规则：
- 项目数据只对团队成员可见
- 财务数据对非财务人员部分隐藏
- 客户联系方式可以设置保密
```

### 8.2 关键校验点

| 模块 | 校验点 | 说明 |
|------|--------|------|
| 客户 | 统一社会信用代码格式 | 18位，有校验规则 |
| 客户 | 身份证号格式和校验位 | 18位，最后一位有校验算法 |
| 项目 | 分配比例之和=100% | 团队成员分配比例 |
| 财务 | 核销金额≤应收金额 | 不能超额核销 |
| 财务 | 提成比例之和≤100% | 案源+办案+律所=100% |
| 证据 | 证据编号案件内唯一 | 如证据1、证据2 |

### 8.3 常见业务边界情况

1. **一个客户多个项目**：正常情况，客户会反复委托
2. **一个项目多次收费**：正常情况，如分期付款
3. **项目团队成员变更**：需要处理已计算提成的历史数据
4. **项目从一审到二审**：可以是同一项目的不同阶段，也可以新建项目
5. **客户付款多于应收**：可能是预付款，需要支持预收款管理
