# 外部业务系统对接档案管理系统指南

本文档面向“律所管理系统 / 案件系统 / 其他业务系统”的接入方，说明如何把归档数据推送到本仓库对应的档案管理系统。

## 1. 对接目标

业务系统在归档完成、结案归档、批量移交等节点，将档案元数据和电子文件下载地址推送给档案管理系统，后者负责：

- 接收归档请求
- 校验来源系统身份
- 下载与存储电子文件
- 建立档案数据与索引
- 在需要时回调处理结果

## 2. 推荐架构

```text
业务系统
  ├─ 结案 / 归档触发
  ├─ 组装档案元数据
  ├─ 提供电子文件 downloadUrl
  └─ 调用 POST /api/open/archive/receive

档案管理系统
  ├─ 校验 X-API-Key
  ├─ 保存档案主数据
  ├─ 下载文件并写入对象存储
  ├─ 记录推送与处理状态
  └─ 按需回调 callbackUrl
```

## 3. 接入前配置

### 3.1 档案系统侧

在档案系统后台“来源管理”中创建来源系统：

- `sourceCode`：建议使用业务系统唯一编码
- `sourceName`：显示名称
- `sourceType`：如 `LAW_FIRM`
- `enabled`：启用

创建成功后保存 API Key。

### 3.2 业务系统侧

至少需要配置：

- 档案系统基础地址，例如 `http://localhost:8090/api`
- `X-API-Key`
- 可被档案系统访问的文件下载地址
- 可选回调地址 `callbackUrl`

## 4. 对接接口

### 4.1 档案接收

- 方法：`POST`
- 路径：`/api/open/archive/receive`
- 认证：`X-API-Key`

最小请求：

```json
{
  "sourceType": "LAW_FIRM",
  "sourceId": "CASE-2026-001",
  "title": "张三诉李四合同纠纷案",
  "archiveType": "DOCUMENT",
  "retentionPeriod": "Y10"
}
```

推荐完整请求：

```json
{
  "sourceType": "LAW_FIRM",
  "sourceId": "CASE-2026-001",
  "sourceNo": "（2026）沪民初字第001号",
  "callbackUrl": "https://lawfirm.example.com/api/archive/callback",
  "async": true,
  "title": "张三诉李四合同纠纷案",
  "archiveType": "DOCUMENT",
  "retentionPeriod": "Y10",
  "responsiblePerson": "王律师",
  "caseNo": "CASE-2026-001",
  "caseName": "张三诉李四合同纠纷案",
  "clientName": "张三",
  "lawyerName": "王律师",
  "caseCloseDate": "2026-03-01",
  "files": [
    {
      "fileName": "起诉状.pdf",
      "fileType": "application/pdf",
      "downloadUrl": "https://lawfirm.example.com/files/1"
    }
  ]
}
```

### 4.2 电子借阅申请

- 方法：`POST`
- 路径：`/api/open/borrow/apply`
- 认证：`X-API-Key`

用于外部系统在业务页面中发起电子借阅链接申请。

### 4.3 健康检查

- 方法：`GET`
- 路径：`/api/open/health`
- 认证：无需

## 5. 字段映射建议

### 5.1 来源标识

- `sourceType`：固定描述来源系统类型，例如 `LAW_FIRM`
- `sourceId`：必须全局唯一，建议使用业务主键
- `sourceNo`：业务编号，可用于人工排查

### 5.2 档案类型

建议统一映射到系统当前可识别的类型，例如：

- `DOCUMENT`
- `SCIENCE`
- `ACCOUNTING`
- `PERSONNEL`
- `SPECIAL`
- `AUDIOVISUAL`

### 5.3 保管期限

常用值：

- `PERMANENT`
- `Y30`
- `Y15`
- `Y10`
- `Y5`

## 6. 文件下载地址要求

- 档案系统必须能直接访问 `downloadUrl`
- 建议使用 HTTPS
- 建议提供至少几分钟以上的有效期
- 单文件大小建议控制在后端上传限制范围内
- 如需鉴权，建议提供可直接访问的短期链接，而不是依赖浏览器 Cookie

## 7. 回调建议

如果配置了 `callbackUrl`：

- 业务系统应实现幂等接收
- 对回调结果做日志记录
- 返回 HTTP 200 表示接收成功

建议回调侧按 `sourceType + sourceId` 定位原业务记录。

## 8. 接入步骤建议

1. 在档案系统创建来源并获取 API Key
2. 从业务系统调用 `GET /api/open/health`
3. 用最小样例请求验证接收成功
4. 增加文件数组、案件字段、回调地址
5. 验证后台档案列表、推送记录、文件存储是否正常

## 9. 常见误区

- 误以为需要签名：当前仓库代码以 `X-API-Key` 为准，不要求签名头
- 误以为 `/open/**` 全部匿名：实际上 `archive/receive` 和 `borrow/apply` 需要 API Key
- 误以为只传业务编号即可：`sourceId` 必须稳定且唯一，不能频繁复用
