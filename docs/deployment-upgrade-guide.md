# 部署与升级手册

本文档面向服务器运维管理员，用于规范律师事务所电子档案系统的首次部署、版本升级、回滚和交付留痕。

## 1. 适用范围

- 单机测试环境
- 单机正式环境
- 面向多个律师事务所客户的标准化交付环境

系统原则：

- 业务管理员只负责系统内配置、档案、借阅、备份恢复。
- 程序部署、镜像升级、数据库迁移、容器重启、异常回滚由运维管理员执行。

## 2. 推荐目录规范

- 源码目录：`/root/src/law-firm-archive`
- 部署目录：`/opt/law-firm-archive`

要求：

- `root/src` 只放源码、构建脚本、版本标签信息
- `/opt/law-firm-archive` 只放部署清单、环境变量、持久化目录、运维台账

## 3. 推荐部署方式

推荐采用 Docker Compose。

推荐原因：

- 对当前规模最直接
- 便于版本化镜像管理
- 便于升级与回滚
- 便于后续复制到多个客户环境

## 4. 首次部署流程

### 4.1 准备源码

```bash
cd /root/src
git clone <your-repository-url> law-firm-archive
cd law-firm-archive
git checkout <release-tag>
```

### 4.2 准备部署目录

```bash
mkdir -p /opt/law-firm-archive
```

需要准备：

- Compose 文件
- `.env` 文件
- 持久化目录
- 反向代理配置
- 版本元数据：`APP_VERSION`、`APP_COMMIT_SHA`、`APP_BUILD_TIME`

### 4.3 初始化数据库

首次部署建议只执行：

- `scripts/init-db/02-schema-consolidated.sql`

不再建议将历史脚本作为生产初始化入口。

### 4.4 拉取镜像并启动

```bash
docker compose pull
docker compose up -d
```

### 4.5 首次部署后必须验证

- 登录正常
- 手工建档正常
- 电子文件上传正常
- 正式归档正常
- 检索正常
- 借阅审批正常
- 备份与恢复页面正常打开

## 5. 标准升级流程

### 5.1 升级前

必须先完成：

- 数据库备份
- 电子文件备份
- 系统配置备份
- 当前版本号、镜像标签、提交号留痕

### 5.2 升级执行

1. 在源码目录拉取目标版本代码
2. 确认目标镜像标签，例如 `v0.1.1`
3. 在部署目录更新 Compose 中的镜像标签与版本元数据
4. 执行 `docker compose pull`
5. 执行 `docker compose up -d`
6. 执行冒烟测试

建议：

- `APP_VERSION` 用于显示产品版本与镜像版本
- `APP_COMMIT_SHA` 用于后台显示对应提交号
- `APP_BUILD_TIME` 用于后台显示本次构建时间

## 6. 标准回滚流程

适用于升级后出现严重故障、无法在短时间内修复的情况。

1. 确认上一稳定版本标签
2. 将 Compose 清单切回上一稳定镜像标签
3. 执行 `docker compose up -d`
4. 如涉及数据库破坏性变更，按备份恢复流程恢复数据库和电子文件
5. 重新执行冒烟测试并记录回滚结论

## 7. 版本号建议

建议统一使用可读发布号：

- `v0.1.0`
- `v0.1.1`
- `v0.2.0`

不建议使用：

- `latest`
- `test`
- `new`

原因：

- 不利于客户环境追踪
- 不利于问题回滚
- 不利于测试台账关联

## 8. 发布留痕要求

每次首次部署、升级、回滚都应至少记录：

- 客户名称
- 环境名称
- 部署版本
- 镜像标签
- Git 提交号
- 执行时间
- 执行人
- 升级前备份编号
- 冒烟测试结果
- 是否回滚

## 9. 面向多客户交付的建议

如果后续要销售给多个律师事务所客户，建议坚持以下原则：

- 每个客户使用独立数据库
- 每个客户使用独立对象存储或独立桶前缀
- 每个客户使用独立 `.env`
- 统一使用同一套镜像与部署模板
- 客户差异通过系统配置、品牌配置和环境变量控制，不改源码分叉
