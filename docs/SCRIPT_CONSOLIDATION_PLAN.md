# 数据库脚本整合评估报告

## 当前状态

| 类别 | 脚本数量 | 文件 |
|------|---------|------|
| **扩展** | 1 | 00-extensions.sql |
| **表结构** | 17 | 01-16 schema + 32 + 50 + 34 |
| **初始化数据** | 2 | 20, 21 |
| **补丁脚本** | 8 | 22-29, 31, 33 |
| **演示数据** | 7 | 30, 40-44, 99 |
| **优化脚本** | 8 | 60-67 |
| **总计** | **46** | - |

## 问题分析

### 1. 脚本编号混乱
- `32-openapi-schema.sql` 放在演示数据之后
- `50-cause-of-action-schema.sql` 放在优化脚本之前
- `34-free-api-integration.sql` 放在演示数据之后

### 2. 功能分类不清晰
- 补丁脚本 (22-23) 实际是冗余的（配置已在20中）
- 演示数据分散在 30, 40-44, 99
- 优化脚本过于细分（8个独立文件）

### 3. 可维护性差
- 添加新功能时难以确定脚本编号
- 依赖关系不够清晰

## 整合方案

### 方案一：激进整合（推荐）

将脚本数量从 46 个减少到 **22 个**：

├── 12-knowledge-schema.sql              # 知识库模块
├── 13-hr-schema.sql                     # 人力资源模块

| **重命名** | 50 | 18 | 移到正确位置 |

### 方案二：保守整合

保持更多细分脚本，仅整理编号：

- 将 32, 34, 50 移到正确位置（重命名为 17, 18, 19）
- 合并演示数据为 30, 31（精简版）, 32（完整版）

## 建议

**推荐采用方案一（激进整合）**，理由：

1. **简化部署**：生产环境只需执行核心脚本
2. **易于维护**：功能分类清晰，新增模块有明确位置
3. **向后兼容**：使用 `IF NOT EXISTS` 确保可重复执行
4. **灵活可选**：演示数据和优化脚本保持独立可选

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 合并后出错影响面大 | 中 | 充分测试，保留原脚本备份 |
| 破坏现有部署流程 | 中 | 提供迁移指南，更新文档 |
| 丢失模块独立性 | 低 | 脚本内保持模块分隔注释 |
