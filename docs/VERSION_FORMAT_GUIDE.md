# 版本号格式使用指南

## 📌 版本号格式说明

系统支持两种版本号格式：

### 1. **简单数字格式**（推荐用于业务版本）
- 格式：`主版本号.次版本号` 或 `主版本号`
- 示例：`0.4`、`1.2`、`2.0`、`3`
- **用途**：用于业务版本标识，简单直观
- **存储位置**：数据库配置表 `sys_config` 中的 `sys.version`

### 2. **语义化版本格式**（用于构建版本）
- 格式：`主版本号.次版本号.修订号[-SNAPSHOT]`
- 示例：`1.0.0`、`1.0.0-SNAPSHOT`、`2.1.3`
- **用途**：用于代码构建和发布管理
- **存储位置**：`pom.xml` 和构建时生成的 `version.properties`

---

## 🎯 版本号优先级

系统会按以下优先级显示版本号：

1. **数据库配置** (`sys.version`) - **优先显示**
   - 支持简单格式：`0.4`、`1.2` 等
   - 可在系统配置页面修改
   - 适合业务版本管理

2. **构建版本** (`pom.xml` 版本)
   - 标准语义化版本格式
   - 仅在数据库版本不存在时使用
   - 适合技术版本管理

## 🔧 如何设置简单版本号（如 0.4）

### 方法1：通过系统配置页面（推荐）

1. 登录系统，进入：**系统管理 > 系统配置 > 通用配置**
2. 找到配置项 `sys.version`
3. 点击"编辑"，修改版本号为简单格式（如：`0.4`）
4. 保存后，版本信息页面会显示新的版本号

### 方法2：通过数据库直接修改

```sql
-- 更新版本号为 0.4
UPDATE sys_config 
SET config_value = '0.4', updated_at = CURRENT_TIMESTAMP 
WHERE config_key = 'sys.version';
```

### 方法3：通过 API 修改

```bash
# 更新版本号为 0.4
curl -X PUT http://localhost:8080/api/system/config/key/sys.version \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d "0.4"
```

## 💡 使用建议

### 简单版本号格式（推荐用于业务）

**适用场景**：
- 业务版本发布（如：0.4 版本发布）
- 用户可见的版本号
- 简单的版本追踪

**示例**：
- `0.1` - 初始版本
- `0.4` - 第4个迭代版本
- `1.0` - 正式版 1.0
- `1.2` - 正式版 1.2
- `2.0` - 大版本 2.0

### 语义化版本格式（用于技术管理）

**适用场景**：
- 代码版本管理
- 依赖管理
- 构建和发布流程

**示例**：
- `1.0.0-SNAPSHOT` - 开发版本
- `1.0.0` - 正式版本
- `1.0.1` - 补丁版本
- `1.1.0` - 功能版本
- `2.0.0` - 主版本

## 🔄 版本号更新流程

### 更新业务版本号（简单格式）

1. **确定新版本号**：如从 `0.3` 升级到 `0.4`
2. **更新数据库配置**：
   ```sql
   UPDATE sys_config 
   SET config_value = '0.4' 
   WHERE config_key = 'sys.version';
   ```
3. **验证显示**：访问版本信息页面，确认显示为 `0.4`

### 更新构建版本号（语义化格式）

1. **确定新版本号**：如从 `1.0.0-SNAPSHOT` 升级到 `1.0.0`
2. **更新 pom.xml**：
   ```xml
   <version>1.0.0</version>
   ```
3. **重新构建**：
   ```bash
   mvn clean package -Pprod
   ```
4. **部署新版本**

## 📝 版本号管理最佳实践

1. **业务版本号**（数据库 `sys.version`）
   - 使用简单格式：`0.4`、`1.2` 等
   - 每次重要功能发布时更新
   - 便于用户理解和记忆

2. **技术版本号**（`pom.xml`）
   - 使用语义化格式：`1.0.0-SNAPSHOT`
   - 遵循语义化版本规范
   - 用于代码管理和依赖控制

3. **版本号对应关系**
   - 建议建立业务版本与技术版本的对应关系
   - 例如：业务版本 `0.4` 对应技术版本 `1.0.0-SNAPSHOT`

4. **版本号记录**
   - 在发布说明中记录版本号
   - 在 Git 标签中使用技术版本号
   - 在用户文档中使用业务版本号

## ❓ 常见问题

### Q: 为什么需要两种版本号格式？

A: 
- **业务版本号**（简单格式）：面向用户，易于理解和记忆
- **技术版本号**（语义化格式）：面向开发，便于代码管理和依赖控制

### Q: 版本号不一致会有问题吗？

A: 不会。系统会优先显示数据库中的业务版本号，技术版本号作为补充信息显示。

### Q: 如何知道当前运行的是哪个版本？

A: 访问 **系统管理 > 系统配置 > 版本信息** 页面，可以看到：
- 应用版本（业务版本号）
- 构建版本（技术版本号）
- Git 提交 ID（精确定位代码版本）
