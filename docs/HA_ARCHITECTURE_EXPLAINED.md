# 高可用架构详解

## 📖 什么是高可用（High Availability）？

**简单理解**：系统在部分组件故障时，仍能继续提供服务，不中断。

**核心概念**：消除单点故障（SPOF - Single Point of Failure）

---

## 🔍 当前架构的单点故障分析

### 当前部署方式：单机 Docker Compose

```
┌─────────────────────────────────────────┐
│         单台服务器 (192.168.50.10)       │
│                                         │
│  ┌──────────┐  ┌──────────┐            │
│  │ Frontend │  │ Backend  │            │
│  │  (1个)   │  │  (1个)   │            │
│  └──────────┘  └──────────┘            │
│                                         │
│  ┌──────────┐  ┌──────────┐            │
│  │PostgreSQL│  │  Redis   │            │
│  │  (1个)   │  │  (1个)   │            │
│  └──────────┘  └──────────┘            │
│                                         │
│  ┌──────────┐  ┌──────────┐            │
│  │  MinIO   │  │OnlyOffice│            │
│  │  (1个)   │  │  (1个)   │            │
│  └──────────┘  └──────────┘            │
└─────────────────────────────────────────┘
```

### ⚠️ 单点故障列表

| 组件 | 故障影响 | 严重程度 |
|------|---------|---------|
| **服务器硬件故障** | 整个系统完全不可用 | 🔴 极高 |
| **PostgreSQL 故障** | 无法读写数据，系统不可用 | 🔴 极高 |
| **Redis 故障** | 缓存失效，会话丢失，性能下降 | 🟠 高 |
| **MinIO 故障** | 无法上传/下载文件 | 🟠 高 |
| **Backend 故障** | API 不可用，前端无法工作 | 🔴 极高 |
| **Frontend 故障** | 用户无法访问界面 | 🟠 高 |
| **OnlyOffice 故障** | 无法编辑文档（可选服务） | 🟡 中 |

## 🎯 高可用方案对比

### 方案一：单机部署（当前）

**特点**：
- ✅ 成本最低（1台服务器）
- ✅ 部署简单
- ❌ 无高可用，单点故障风险高
- ❌ 无法水平扩展

**适用场景**：
- 小型律所（< 50人）
- 对可用性要求不高
- 预算有限

---

### 方案二：Docker Swarm 多节点（推荐）

**架构**：

```
┌─────────────────────────────────────────────────────────┐
│              负载均衡 (Swarm Ingress)                    │
│                   端口: 80, 443                          │
└─────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  节点1       │  │  节点2       │  │  节点3       │
│ 192.168.50.10│  │ 192.168.50.11│  │ 192.168.50.12│
│              │  │              │  │              │
│ Frontend × 2 │  │ Frontend × 2 │  │ Frontend × 2 │
│ Backend × 2  │  │ Backend × 2  │  │ Backend × 2  │
└──────────────┘  └──────────────┘  └──────────────┘
        │                 │                 │
        └─────────────────┼─────────────────┘
                          ▼
              ┌───────────────────────┐
              │  共享存储 (NFS/NAS)    │
              │  - PostgreSQL (主从)  │
              │  - Redis (哨兵模式)    │
              │  - MinIO (分布式)      │
              └───────────────────────┘
```

**特点**：
- ✅ 应用层高可用（Frontend/Backend 多实例）
- ✅ 自动故障转移
- ✅ 可以水平扩展
- ⚠️ 需要共享存储
- ⚠️ 数据库需要主从配置

**适用场景**：
- 中型律所（50-200人）
- 需要7×24小时可用
- 有一定预算

---

### 方案三：Kubernetes（企业级）

**特点**：
- ✅ 完整的高可用方案
- ✅ 自动扩缩容
- ✅ 滚动更新，零停机
- ❌ 复杂度高，需要专业运维
- ❌ 资源消耗大

**适用场景**：
- 大型律所（> 200人）
- 有专业运维团队
- 预算充足

---

## 🔧 实现高可用的关键点

### 1. 应用层高可用

**问题**：Backend 只有一个实例，挂了就全挂

**解决**：运行多个 Backend 实例
```yaml
# Docker Swarm 配置
backend:
  deploy:
    replicas: 3  # 运行 3 个实例
```

**效果**：
- 1个 Backend 挂了，还有 2 个可以服务
- 负载自动分配到健康的实例

---

### 2. 数据库高可用

**问题**：PostgreSQL 只有一个，数据丢失风险高

**解决方案 A：主从复制（推荐）**
```
主库 (Master) ──复制──> 从库 (Slave)
    │                        │
    └──写入                  └──只读
```

- ✅ 主库故障时，从库可以提升为主库
- ✅ 数据有备份
- ⚠️ 需要配置自动切换（如 Patroni）

**解决方案 B：云数据库（最简单）**
- 使用阿里云 RDS / 腾讯云 PostgreSQL
- ✅ 自动备份
- ✅ 高可用配置
- ✅ 专业运维

---

### 3. 缓存高可用

**问题**：Redis 单点故障

**解决**：Redis 哨兵模式（Sentinel）
```
Redis Master ──复制──> Redis Slave
     │                        │
     └─── Sentinel 监控 ─────┘
```

- ✅ 主节点故障时，自动切换到从节点
- ✅ 数据不丢失

---

### 4. 存储高可用

**问题**：MinIO 单点，文件丢失风险

**解决**：MinIO 分布式模式
```yaml
# MinIO 分布式（4个节点，每个节点4块盘）
minio:
  command: server http://node{1...4}/data{1...4}
```

- ✅ 数据冗余（允许部分节点故障）
- ✅ 自动修复
- ⚠️ 需要多台服务器或多块硬盘

---

### 5. 负载均衡

**问题**：如何将请求分发到多个实例？

**Docker Swarm 自动提供**：
- ✅ 内置负载均衡
- ✅ 健康检查
- ✅ 自动故障转移

**Nginx 负载均衡**（可选）：
```nginx
upstream backend {
    server backend1:8080;
    server backend2:8080;
    server backend3:8080;
}
```

## 📊 高可用级别对比

| 方案 | 可用性 | 成本 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| **单机部署** | 99% | 低 | 低 | 小型律所 |
| **Swarm 多节点** | 99.9% | 中 | 中 | 中型律所 |
| **Kubernetes** | 99.99% | 高 | 高 | 大型律所 |

**可用性说明**：
- 99% = 每年约 87.6 小时不可用
- 99.9% = 每年约 8.76 小时不可用
- 99.99% = 每年约 52.56 分钟不可用

## 🚀 推荐的高可用实施路径

### 阶段一：基础高可用（低成本）

1. **数据库备份**
   - 定期自动备份 PostgreSQL
   - 备份文件存储到 NAS 或云存储

2. **应用层多实例**
   - 使用 Docker Swarm，运行 2-3 个 Backend 实例
   - 运行 2 个 Frontend 实例

3. **监控告警**
   - Prometheus + Grafana 监控
   - 服务故障时发送告警

**成本**：增加 1-2 台服务器（或使用现有服务器）

---

### 阶段二：完整高可用（中等成本）

1. **数据库主从**
   - PostgreSQL 主从复制
   - 自动故障转移

2. **Redis 哨兵**
   - Redis 主从 + Sentinel

3. **MinIO 分布式**
   - 多节点 MinIO 集群

**成本**：3-5 台服务器

---

### 阶段三：企业级高可用（高成本）

1. **Kubernetes 集群**
   - 完整的容器编排
   - 自动扩缩容

2. **云数据库**
   - 使用云服务商的托管数据库

3. **CDN + 多地域部署**
   - 全球加速
   - 多地域容灾

**成本**：10+ 台服务器或云服务费用

---

## 💡 总结

**高可用的本质**：
1. **冗余**：关键组件有备份
2. **监控**：及时发现问题
3. **自动恢复**：故障时自动切换
4. **数据安全**：数据不丢失

**你的系统需要多高的可用性？**
- 如果只是内部使用，单机 + 定期备份可能就够了
- 如果需要 7×24 小时服务，建议至少 Swarm 多节点
- 如果是关键业务，建议 Kubernetes + 云数据库
