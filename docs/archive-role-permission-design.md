# 电子档案系统角色与权限设计

## 1. 目的

本设计文档用于明确律师事务所电子档案系统的角色边界、业务流程、权限矩阵和后续开发约束，避免后续开发过程中出现：

- 业务角色定义反复变动
- 菜单、页面、按钮、接口权限不一致
- 无权限用户仍看到不应显示的导航
- 普通用户直接把档案塞进正式档案库

本设计优先服务于律师事务所电子档案真实业务，不以技术实现方便为第一目标。

## 2. 业务前提

本系统是**电子档案系统**，不是单纯纸质档案台账。

核心业务链路分三段：

1. 入库
2. 保存
3. 借阅利用

业务原则：

- 电子档案优先，纸质位置为辅
- 普通用户可提交档案材料，但不能直接让档案正式入库
- 入库必须经过审核
- 入库后由档案管理员负责整理、补充、利用
- 借阅利用必须受控，包含纸质借阅、电子浏览、电子下载

## 3. 角色定义

系统最终采用 4 类角色，不再继续扩散角色。

### 3.1 系统管理员 `SYSTEM_ADMIN`

职责：

- 完全权限
- 管理用户、角色、系统配置、备份恢复、日志
- 可查看和处理全部档案
- 可介入审核、整理、借阅等全部业务链路
- 负责系统级维护与配置口径

说明：

- 系统管理员拥有覆盖权限，但不意味着日常业务都由其处理
- 业务上应尽量由审核员和档案管理员承担各自职责

### 3.2 档案审核员 `ARCHIVE_REVIEWER`

职责：

- 专门处理入库前审核
- 审核普通用户提交的档案材料和入库申请
- 可通过、退回、填写审核意见
- 只负责“是否准入库”，不负责正式归档后的整理利用

说明：

- 审核员不负责系统设置
- 审核员不负责借阅管理
- 审核员不负责正式档案整理

### 3.3 档案管理员 `ARCHIVE_MANAGER`

职责：

- 负责已入库档案的整理、补充、保存和利用
- 负责目录修整、电子文件补充、纸质位置维护
- 负责借阅审批、借出、归还、借阅链接和利用控制

约束：

- 可以补充正式入库后的档案内容
- 不能删除非补充性质的原始入库内容
- 不能修改系统级配置和用户权限

### 3.4 普通用户 `USER`

职责：

- 律师、律师助理等业务人员
- 提交档案入库申请
- 上传电子档案材料
- 发起纸质借阅申请
- 发起电子浏览申请、电子下载申请

约束：

- 上传材料不等于正式入库
- 无权直接批准入库
- 无权直接修改正式档案的核心内容
- 无权进入系统管理后台

## 4. 标准业务流程

### 4.1 档案入库流程

1. 普通用户完成项目后，整理待归档材料
2. 普通用户上传电子档案材料并提交入库申请
3. 系统通知档案审核员有待审核档案
4. 档案审核员审核：
   - 通过：档案正式入库
   - 退回：普通用户补正后重新提交
5. 档案正式入库后，进入档案管理员职责范围
6. 档案管理员继续整理、补充、规范目录与位置

### 4.1.1 管理系统推送入库规则

管理系统推送过来的档案，不采用“一律人工审批”或“一律直接入库”的简单模式，而是按**档案来源预设策略**执行。

规则如下：

1. 系统必须预先配置档案来源
2. 每个来源必须明确其入库策略
3. 入库策略分为两类：
   - 直接落库
   - 提交审核

具体逻辑：

- 如果推送来源被配置为“直接落库”，则该来源推送过来的档案可直接进入正式入库流程，无须审核员审批
- 如果推送来源未配置为“直接落库”，则推送档案进入待审核状态，由档案审核员接管，审核通过后才能正式入库

补充要求：

- 无论是否直接落库，都必须保留推送记录、来源标识和审计日志
- 直接落库来源应仅限可信来源系统
- 若后续来源策略被调整，只影响新推送数据，不自动改变既有档案状态
- 推送失败、字段不完整、附件异常时，即使来源为“直接落库”，也应转入异常记录或待人工处理队列，不得静默入库

### 4.2 档案保存流程

1. 档案管理员检查已入库档案的目录和电子文件完整性
2. 补充保管期限、分类、全宗、位置等信息
3. 如存在后补材料，按补充方式追加，不覆盖原始入库内容
4. 确保电子档案可检索、可预览、可受控利用

### 4.3 借阅利用流程

1. 普通用户发起借阅申请
2. 借阅类型分为：
   - 纸质借阅
   - 电子浏览
   - 电子下载
3. 档案管理员负责审批和借阅流转控制
4. 电子借阅可通过受控预览或外链实现
5. 纸质借阅需记录借出、归还、逾期等状态

## 5. 权限设计原则

权限必须统一控制在四层：

1. 菜单可见性
2. 页面路由访问
3. 页面按钮/操作可见性
4. 后端接口权限

统一原则：

- 无权限用户看不到对应菜单
- 无权限用户看不到对应按钮
- 无权限用户强行输入地址时返回 `403`
- 后端接口始终是最终安全边界
- 前端隐藏不等于后端放开

## 7. 页面操作权限矩阵

### 7.1 普通用户

允许：

- 新建入库申请
- 上传档案材料
- 查看自己有权限的档案
- 发起纸质借阅申请
- 发起电子浏览申请
- 发起电子下载申请

禁止：

- 审核入库
- 修改正式档案核心信息
- 删除正式档案内容
- 管理借阅链接
- 管理系统配置

### 7.2 档案审核员

允许：

- 查看待审核档案
- 审核通过
- 审核退回
- 填写审核意见
- 查看审核记录

禁止：

- 正式档案整理和补充
- 借阅审批
- 系统管理配置

### 7.3 档案管理员

允许：

- 整理正式档案
- 补充目录和电子文件
- 维护纸质位置
- 处理借阅审批
- 管理借出、归还、逾期
- 生成和回收借阅链接

禁止：

- 删除原始入库内容
- 处理系统配置、用户权限、备份恢复

### 7.4 系统管理员

允许：

- 全部操作

## 8. 状态流转设计

现有系统状态需按业务重新梳理，建议引入“入库审核”相关状态。

建议档案主状态：

- `DRAFT`：草稿 / 未提交
- `PENDING_REVIEW`：待审核
- `REJECTED`：审核退回
- `APPROVED`：审核通过，准入库
- `CATALOGING`：整理中
- `STORED`：已归档
- `BORROWED`：借出中
- `DESTROYED`：已销毁

建议说明：

- 普通用户提交后：进入 `PENDING_REVIEW`
- 推送来源为“需审核”时：进入 `PENDING_REVIEW`
- 推送来源为“直接落库”时：可直接进入 `APPROVED` 或 `CATALOGING`
- 审核通过后：进入 `APPROVED` 或直接进入 `CATALOGING`
- 档案管理员整理完成后：进入 `STORED`

## 9. 数据修改约束

为了满足“档案管理员只能补充、不能删除非补充内容”的业务要求，正式开发时必须增加以下约束：

### 9.1 原始内容保护

原始入库内容包括：

- 初始目录
- 初始电子文件
- 初始入库申请信息

约束：

- 档案管理员不得直接物理删除原始内容
- 如需修正，应通过补充记录、修订记录、版本记录等方式处理

### 9.2 补充内容标记

建议增加字段或标记：

- 是否原始入库内容
- 是否补充内容
- 补充人
- 补充时间
- 补充说明

## 10. 前端开发约束

### 10.1 菜单

- 左侧菜单必须按角色动态裁剪
- 无权限菜单不显示
- 不能依赖点击后 403 充当前端权限控制

### 10.2 路由

- 路由 `meta.roles` 必须与角色矩阵一致
- 普通用户输入受限地址时应进入 `403`

### 10.3 页面组件

- 页面初始化时不得请求当前角色无权访问的接口
- 例如普通用户不得自动请求系统管理员专属接口

### 10.4 帮助中心

- 帮助中心内容必须按新 4 角色重写
- 不再保留安全管理员、审计管理员等旧角色口径

## 11. 后端开发约束

### 11.1 接口权限

- `@PreAuthorize` 必须按新 4 角色统一调整
- 控制器、服务、查询范围必须一致

### 11.2 数据范围

- 普通用户只应看到自己有权访问的档案和借阅数据
- 档案审核员只应看到待审核范围和审核记录
- 档案管理员应看到档案管理和借阅管理范围
- 系统管理员可看全部

### 11.3 审核链路

必须新增或重构：

- 入库申请表 / 入库审核记录
- 审核通过 / 退回接口
- 审核意见字段
- 审核通知机制
- 推送来源审批策略字段
- 推送来源是否“直接落库”的配置能力
- 推送异常转人工处理机制

## 12. 当前代码差距

根据当前代码现状，已知差距包括：

- 当前角色仍是旧模型：
  - `SYSTEM_ADMIN`
  - `SECURITY_ADMIN`
  - `AUDIT_ADMIN`
  - `ARCHIVIST`
  - `USER`
- 当前尚未形成正式权限矩阵
- 当前仍存在部分菜单、页面、接口权限不一致
- 当前普通用户上传与正式入库之间缺少清晰审核闭环
- 当前“档案管理员不能删除原始内容”尚未形成强约束

## 13. 开发顺序建议

建议按以下顺序实施，不要跳步：

1. 先调整角色常量与角色文案
2. 设计并实现入库审核状态流转
3. 调整前端菜单权限矩阵
4. 调整前端路由权限矩阵
5. 调整页面级按钮权限
6. 调整后端接口 `@PreAuthorize`
7. 增加原始内容保护与补充内容标记
8. 重写帮助中心与用户手册口径
9. 完成普通用户 / 审核员 / 档案管理员 / 系统管理员四角色回归测试

## 14. 验收标准

满足以下条件后，才可认为该权限设计落地完成：

- 普通用户登录后看不到管理类菜单
- 普通用户提交档案后不会直接正式入库
- 审核员可以看到待审核档案并完成通过/退回
- 档案管理员可以补充正式档案，但不能删除原始入库内容
- 系统管理员拥有完整后台权限
- 强输 URL 时，无权限页面统一返回 `403`
- 前端不会自动请求当前角色无权访问的接口
- 四类角色均完成冒烟测试并记录台账
