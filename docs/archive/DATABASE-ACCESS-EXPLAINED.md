# 数据库访问方式说明

## 🤔 问题：数据库不暴露端口是什么意思？

### 当前配置

```yaml
# docker-compose.prod.yml
postgres:
  # 没有 ports 配置
  # 意味着：数据库端口 5432 不暴露到宿主机
  # 只有 Docker 内部网络可以访问
```

**这意味着什么？**
- ✅ 应用（backend）可以访问数据库（通过 Docker 内部网络）
- ✅ 用户通过应用正常使用系统（完全不受影响）
- ❌ 外部无法直接连接数据库（更安全）

---

## 👥 用户规模 vs 数据库访问

### 重要理解：普通用户不需要直接访问数据库

```
┌─────────────────────────────────────────┐
│  普通用户（几十上百人）                   │
│  ↓                                       │
│  通过浏览器访问前端                       │
│  ↓                                       │
│  前端调用后端 API                        │
│  ↓                                       │
│  后端访问数据库（Docker 内部）           │
└─────────────────────────────────────────┘
```

**普通用户：**
- 通过浏览器使用系统
- 完全不需要知道数据库存在
- 不需要直接访问数据库

**只有运维人员需要访问数据库：**
- 数据库备份
- 数据迁移
- 故障排查
- 性能优化

---

## 🔐 数据库不暴露端口的好处

### 1. 安全性

**暴露端口的风险：**
```bash
# 如果暴露端口：
postgres:
  ports:
    - "5432:5432"  # ← 任何人都可以尝试连接

# 攻击者可以：
# 1. 扫描端口，发现数据库
# 2. 尝试暴力破解密码
# 3. 如果密码泄露，直接访问数据库
```

**不暴露端口：**
```bash
# 不暴露端口：
postgres:
  # ports:  # ← 不暴露
  # 攻击者无法扫描到数据库端口
  # 即使知道密码，也无法连接
```

### 2. 用户规模不影响

**无论用户多少，都不应该暴露数据库端口：**

| 用户规模 | 数据库访问方式 | 是否需要暴露端口 |
|---------|--------------|----------------|
| 10人 | 通过应用访问 | ❌ 不需要 |
| 100人 | 通过应用访问 | ❌ 不需要 |
| 1000人 | 通过应用访问 | ❌ 不需要 |
| 10000人 | 通过应用访问 | ❌ 不需要 |

**原因：**
- 普通用户通过应用访问，不直接连接数据库
- 用户数量不影响数据库端口暴露的必要性
- 只有运维人员需要直接访问数据库

## 🛠️ SSH 隧道：运维人员访问数据库的方式

### 什么是 SSH 隧道？

**简单理解：**
- 通过 SSH 连接服务器
- 在本地创建一个"隧道"
- 通过隧道访问服务器上的数据库

**示意图：**
```
运维人员电脑
    ↓ SSH 连接
服务器（SSH 端口 22）
    ↓ 隧道转发
数据库（端口 5432，仅内部访问）
```

### 如何使用？

#### 方法 1：SSH 端口转发（推荐）

```bash
# 在运维人员电脑上执行：
ssh -L 5432:localhost:5432 user@your-server-ip

# 然后在本地的数据库工具中连接：
# 主机：localhost
# 端口：5432
# 用户名：law_admin
# 密码：your-db-password
```

**解释：**
- `-L 5432:localhost:5432`：将本地 5432 端口转发到服务器的 5432 端口
- 连接 `localhost:5432` 实际上连接的是服务器的数据库

#### 方法 2：使用数据库管理工具（图形界面）

**DBeaver（推荐）：**
1. 创建新连接
2. 选择 PostgreSQL
3. 在 "SSH" 标签页配置：
   - SSH Host: `your-server-ip`
   - SSH Port: `22`
   - SSH User: `your-ssh-user`
   - SSH Password: `your-ssh-password`
4. 在主标签页配置：
   - Host: `localhost`（因为通过 SSH 隧道）
   - Port: `5432`
   - Database: `law_firm`
   - User: `law_admin`
   - Password: `your-db-password`

**Navicat：**
1. 创建连接
2. 选择 "SSH" 标签
3. 配置 SSH 信息
4. 配置数据库信息

#### 方法 3：命令行工具

```bash
# 1. 建立 SSH 隧道（后台运行）
ssh -f -N -L 5432:localhost:5432 user@your-server-ip

# 2. 连接数据库
psql -h localhost -p 5432 -U law_admin -d law_firm

# 3. 执行 SQL
SELECT COUNT(*) FROM users;
```

## 📋 实际使用场景

### 场景 1：日常运维（不频繁）

**频率：** 每周几次或每月几次

**操作：**
```bash
# 1. 建立 SSH 隧道
ssh -L 5432:localhost:5432 user@server

# 2. 使用数据库工具连接（DBeaver、Navicat等）
# 或使用命令行工具

# 3. 完成后关闭 SSH 连接
```

**时间：** 每次操作只需几秒钟建立连接

### 场景 2：数据库备份（定期）

**频率：** 每天或每周自动备份

**操作：**
```bash
# 通过 SSH 执行备份命令（不需要端口转发）
ssh user@server "docker exec law-firm-postgres pg_dump -U law_admin law_firm > backup.sql"

# 或使用备份脚本
./scripts/db-auto-backup.sh
```

**说明：** 备份脚本在服务器上执行，不需要暴露端口

### 场景 3：故障排查（紧急情况）

**频率：** 很少发生

**操作：**
```bash
# 1. SSH 到服务器
ssh user@server

# 2. 进入数据库容器
docker exec -it law-firm-postgres psql -U law_admin -d law_firm

# 3. 执行排查命令
SELECT * FROM pg_stat_activity;
```

**说明：** 在服务器上直接操作，不需要暴露端口

## ✅ 总结

### 对于你的项目（几十上百人使用）

**数据库配置：**
```yaml
# docker-compose.prod.yml
postgres:
  # 不暴露端口 ✅
  # ports:  # ← 保持这样，不暴露
```

**为什么？**
1. ✅ **用户不受影响**：普通用户通过应用正常使用，完全感觉不到
2. ✅ **更安全**：数据库不暴露到公网，降低被攻击风险
3. ✅ **运维方便**：需要时通过 SSH 隧道访问，操作简单
4. ✅ **标准做法**：这是生产环境的标准安全配置

### 运维人员访问数据库

**方式：**
- 通过 SSH 隧道（推荐）
- 或 SSH 到服务器后直接操作

**频率：**
- 不频繁（每周几次或更少）
- 每次操作只需几秒钟建立连接

**工具：**
- DBeaver（图形界面，推荐）
- Navicat（图形界面）
- psql（命令行）

### 关键点

1. **普通用户**：完全不需要知道数据库存在，通过应用正常使用
2. **运维人员**：需要时通过 SSH 隧道访问，操作简单
3. **安全性**：数据库不暴露端口，更安全
4. **用户规模**：无论用户多少，都不应该暴露数据库端口

**保持当前配置（数据库不暴露端口）**

**原因：**
- ✅ 这是生产环境的标准安全配置
- ✅ 用户使用完全不受影响
- ✅ 运维需要时通过 SSH 隧道访问，操作简单
