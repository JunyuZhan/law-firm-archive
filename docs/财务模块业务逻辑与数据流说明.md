# 财务模块业务逻辑与数据流说明

## 1. 模块概述

财务模块是律所管理系统的核心业务模块，负责合同管理、收款管理、提成计算和财务统计。该模块采用**事件驱动架构**，通过监听合同审批事件自动同步数据，实现财务数据的自动化处理。

### 1.1 核心功能

- **合同管理**：创建、编辑、审批合同，自动同步到财务和行政模块
- **收款管理**：登记收款、确认收款、锁定数据、变更申请
- **提成计算**：基于收款自动计算律师提成（三层分配模型）
- **财务统计**：收款进度、提成汇总、财务报表

### 1.2 核心角色

| 角色 | 权限 | 职责 |
|------|------|------|
| LEAD_LAWYER（承办律师） | 查看自己的收款和提成 | 了解案件收款进度 |
| FINANCE_MANAGER（财务主管） | 管理所有收款、提成审批 | 审批提成、发放提成 |
| FINANCE（财务） | 登记收款、确认收款 | 日常收款操作 |
| DIRECTOR（主任） | 提成审批、财务汇总、修改提成规则 | 审批提成、查看财务数据、调整提成设定 |
| ADMIN（系统管理员） | 全部权限、修改提成规则 | 系统配置、调整提成设定 |

## 2. 核心业务流程

### 2.1 合同审批 → 财务同步流程

```
合同创建
  ↓
合同审批通过 → 发布 ContractApprovedEvent
  ↓
ContractSyncService 监听事件
  ├─ 同步到财务模块
  │  └─ 自动创建 Fee 记录（收费记录）
  │     ├─ 如果有付款计划 → 按计划创建多条 Fee（分期收款）
  │     └─ 如果无付款计划 → 创建一条总额 Fee
  │
  └─ 同步到行政模块
     └─ 记录合同基本信息
```

**关键点**：
- 合同审批通过后，自动在财务模块创建收费记录
- 收费记录状态初始为 `PENDING`（待收款）
- 如果合同有付款计划，会按计划创建多条分期收费记录

### 2.2 收款登记 → 确认 → 提成计算流程

```
财务人员登记收款
  ↓
创建 Payment 记录（状态：PENDING）
  ↓
财务人员确认收款
  ↓
Payment 状态变更为 CONFIRMED
  ├─ 自动锁定 Payment 记录（locked=true）
  ├─ 更新 Fee 的已收金额（paidAmount）
  ├─ 更新 Fee 状态（PENDING → PARTIAL → PAID）
  ├─ 更新 Contract 的已收金额
  │
  └─ 【自动触发】CommissionCalculationService.calculateCommission()
     ├─ 获取合同参与人（ContractParticipant）
     ├─ 查找适用的提成规则（CommissionRule）
     ├─ 执行三层分配计算
     │  ├─ 第一层：律所留存（firmRetentionRate）
     │  ├─ 第二层：案源提成（originatorRate）
     │  └─ 第三层：办案提成（剩余部分按分配比例）
     └─ 生成 Commission 记录
```

**关键点**：
- 收款确认后自动锁定，防止误修改
- 已锁定的收款如需修改，必须通过变更申请流程
- 提成计算自动触发，无需手动操作

### 2.3 提成计算详解（三层分配模型）

#### 2.3.1 数据来源

提成计算基于**财务模块自己的数据**：
- 收款数据：`finance_payment` 表
- 合同数据：`finance_contract` 表
- 参与人数据：`contract_participant` 表（财务模块维护）
- 用户薪酬模式：`sys_user` 表（系统模块）

#### 2.3.2 三层分配算法

假设收款金额为 100 万，规则配置如下：
- 律所留存率：30%
- 案源提成率：20%（占剩余部分）
- 办案人员分配比例：LEAD 60%，CO_COUNSEL 40%

**计算过程**：

```
第一层 - 律所留存
  收款金额：1,000,000
  律所留存：1,000,000 × 30% = 300,000
  剩余：700,000

第二层 - 案源提成
  案源提成总额：700,000 × 20% = 140,000
  剩余：560,000
  
  案源人分配（假设有1个案源人）：
    案源人：140,000 × 100% = 140,000

第三层 - 办案提成
  办案提成总额：560,000
  
  办案人分配（假设LEAD和CO_COUNSEL各1人）：
    LEAD（60%）：560,000 × 60% = 336,000
    CO_COUNSEL（40%）：560,000 × 40% = 224,000
```

#### 2.3.3 薪酬模式过滤

系统支持两种薪酬模式：
- **提成制**（COMMISSION）：参与提成分配
- **授薪制**（SALARIED）：不参与提成分配

**处理逻辑**：
1. 获取合同参与人列表
2. 过滤出提成制律师（排除授薪制）
3. 对提成制律师重新归一化分配比例
4. 为授薪制律师生成 0 提成记录（用于审计）

**示例**：
```
原始参与人：
  - LEAD（提成制）：分配比例 60%
  - CO_COUNSEL（授薪制）：分配比例 40%

过滤后：
  - LEAD（提成制）：分配比例 100%（60% ÷ 60% = 100%）
  - CO_COUNSEL（授薪制）：生成 0 提成记录
```

#### 2.3.4 阶梯费率

提成规则支持阶梯费率配置，根据收款金额自动应用不同的提成比例。

**配置示例**（JSON格式）：
```json
[
  {
    "minAmount": 0,
    "maxAmount": 100000,
    "rate": 0.8
  },
  {
    "minAmount": 100000,
    "maxAmount": 500000,
    "rate": 0.85
  },
  {
    "minAmount": 500000,
    "maxAmount": null,
    "rate": 0.9
  }
]
```

**应用规则**：
- 收款 50,000 → 应用 0.8 比例
- 收款 200,000 → 应用 0.85 比例
- 收款 1,000,000 → 应用 0.9 比例

## 4. 关键业务规则

### 4.1 收款锁定规则

| 状态 | 是否锁定 | 可操作 | 说明 |
|------|--------|--------|------|
| PENDING | 否 | 修改、删除、确认 | 待确认状态，可直接修改 |
| CONFIRMED | 是 | 变更申请 | 已确认状态，禁止直接修改 |
| CANCELLED | 是 | 无 | 已取消状态，不可操作 |

### 4.2 收费状态流转

```
PENDING (待收款)
  ↓ 收款确认
PARTIAL (部分收款) ← 继续收款
  ↓ 全额收款
PAID (已收款)
```

### 4.3 提成状态流转

```
CALCULATED (已计算)
  ↓ 主任审批
APPROVED (已审批)
  ↓ 财务发放
PAID (已发放)
```

### 4.4 变更申请流程

已锁定的收款如需修改，必须通过变更申请流程：

```
提交变更申请
  ↓
创建审批流程（自动分配给财务主管）
  ↓
财务主管审批
  ├─ 批准 → 更新收款金额 → 更新收费和合同数据
  └─ 拒绝 → 变更申请被拒绝
```

## 7. 常见问题

### Q1: 为什么收款确认后不能直接修改？

**A**: 为了保证财务数据的准确性和可追溯性。收款确认后会自动锁定，防止误操作。如需修改，必须通过变更申请流程，由财务主管审批，确保所有变更都有记录。

### Q2: 提成是如何自动计算的？

**A**: 当财务人员确认收款时，系统会自动触发 `CommissionCalculationService.calculateCommission()` 方法，根据合同参与人、提成规则和收款金额，自动计算提成并生成提成记录。

### Q3: 授薪制律师为什么会生成 0 提成记录？

**A**: 为了保证审计的完整性。即使授薪制律师不参与提成分配，系统也会为其生成 0 提成记录，记录其薪酬模式和不参与分配的原因，便于后续审计和查询。

### Q4: 如何处理多个案源人的情况？

**A**: 系统会根据 `contract_participant` 表中的 `commission_rate` 字段，按比例分配案源提成。例如，如果有两个案源人，分配比例分别为 60% 和 40%，则案源提成总额会按这个比例分配给两个案源人。

### Q5: 阶梯费率如何应用？

**A**: 提成规则支持配置多个阶梯，系统会根据收款金额自动匹配相应的阶梯，应用对应的提成比例。例如，收款 50 万应用 85% 比例，收款 100 万应用 90% 比例。

### Q6: 谁可以修改提成规则？

**A**: 只有系统管理员（ADMIN）和主任（DIRECTOR）可以修改提成规则。这两个角色可以在后期根据律所实际情况调整律所留存率、案源提成率、税率、管理费率、阶梯费率等参数。修改后的规则对新产生的收款自动生效。

## 8. 开发指南

### 8.1 添加新的提成规则

1. 系统管理员或主任通过后台管理界面访问提成规则管理页面
2. 点击"新增规则"，配置规则参数：
   - 律所留存率
   - 案源提成率
   - 税率
   - 管理费率
   - 阶梯费率等
3. 设置生效日期和失效日期
4. 标记为默认规则（可选）
5. 保存后对新产生的收款自动生效

**权限要求**：`finance:commission:rule:manage`（DIRECTOR 或 ADMIN）

### 8.2 修改提成计算逻辑

1. 编辑 `CommissionCalculationService.calculateThreeTierCommission()` 方法
2. 修改分配算法或添加新的分配层级
3. 更新相关的单元测试
4. 更新本文档

### 8.3 添加新的财务报表

1. 在 `CommissionAppService` 中添加新的查询方法
2. 在 `CommissionRepository` 中添加对应的 SQL 查询
3. 在前端创建新的报表页面
4. 配置相应的权限

## 9. 性能优化建议

1. **收款查询**：为 `finance_payment` 表的 `status` 和 `contract_id` 字段添加索引
2. **提成查询**：为 `finance_commission` 表的 `user_id` 和 `status` 字段添加索引
3. **合同同步**：使用异步处理，避免阻塞合同审批流程
4. **提成计算**：对大批量收款使用批量计算，减少数据库查询次数
