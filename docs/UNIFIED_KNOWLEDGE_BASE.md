## 1. 律所业务双维度模型 (核心概念)

> [!IMPORTANT]
> **绝对不能混淆**：项目的“角色”和团队成员的“薪酬模式”是两回事。

### 维度一：项目角色 (你干了什么活)
- **主办律师 (Lead)**: 每个项目唯一，负总责，拥有分配团队比例的权限。
- **协办律师 (Associate)**: 协助办案，0-N个。
- **案源人 (Originator)**: 负责带入客户，不一定参与办案，但参与第一波提成分配。

### 维度二：薪酬模式 (你怎么领工资)
- **提成制律师 (Commission)**: 只有办了案子、回了款才有钱拿（或极低底薪）。
- **授薪制律师 (Salaried)**: 拿固定月薪。**重点**：系统中授薪律师做承办也不拿提成。
- **混合制律师**: 底薪 + 提成。

## 2. 财务管理：三层分配算法 (核心代码逻辑)

系统在“收款确认”的那一刻，会自动触发以下提成逻辑：

1.  **第一层：律所留存 (Firm Retention)**
    - 计算公式：`收款金额 * 律所固定比例`
    - 处理：这部分钱归律所，不分给个人。
2.  **第二层：案源提成 (Originator Fee)**
    - 计算公式：`(收款金额 - 律所留存) * 案源规则比例`
    - 处理：分给对应的案源人。
3.  **第三层：办案提成 (Practice Fee)**
    - 计算公式：`剩余全部金额`
    - 处理：按主办律师在 `MatterTeam` 中设置的比例，分给所有**提成制**办案律师。
    - **归一化处理**：如果团队中有授薪制律师，其份额会自动扣除，其余金额在提成律师间重新按权重拉平。

## 3. 文书印章：状态机与审计

### 文书状态流转
- `草稿` -> `申请用印` -> `行政审批` -> `已盖章` -> `归档`。
- **注意**: 目前采用手动状态机驱动，非 Flowable 引擎。

### 审计机制
- 系统所有文书访问均通过 `DocAccessLogService` 自动切面记录：**时间、地点、人、动作、IP**。

## 5. 开发者规范
- **代码分层**: `interfaces` (Controller) -> `application` (Service) -> `domain` (Entity/Repository) -> `infrastructure` (Util/DB)。
- **错误处理**: 统一使用 `BusinessException`，严禁在 Controller 捕获异常。
- **数据一致性**: 跨模块调用优先使用 `ApplicationEvent` (Spring Event)。
