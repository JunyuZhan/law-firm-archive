## 二、模块开发规范

### 2.1 每个业务模块的标准结构

以 `客户管理(client)` 模块为例：

```
domain/client/
├── entity/                   # 领域实体
│   ├── Client.java           # 客户（@SuperBuilder继承BaseEntity）
│   └── ConflictCheck.java    # 利冲审查
├── repository/               # 仓储类（继承AbstractRepository）
│   ├── ClientRepository.java
│   └── ConflictCheckRepository.java
└── service/                  # 领域服务（可选，复杂业务逻辑）
    └── ConflictCheckDomainService.java

application/client/
├── service/                  # 应用服务
│   ├── ClientAppService.java
│   └── ConflictCheckAppService.java
├── dto/                      # 数据传输对象
│   ├── ClientDTO.java
│   ├── ClientQueryDTO.java   # 查询参数
│   └── ConflictCheckDTO.java
├── command/                  # 命令对象（写操作）
│   ├── CreateClientCommand.java
│   ├── UpdateClientCommand.java
│   └── ApplyConflictCheckCommand.java
└── assembler/                # 转换器（可选）
    └── ClientAssembler.java

interfaces/rest/client/
├── ClientController.java     # 客户接口
└── ConflictCheckController.java

infrastructure/persistence/mapper/
├── ClientMapper.java         # Mapper接口（注解SQL + 复杂查询）
└── ConflictCheckMapper.java
```

### 2.2 命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 实体 | 名词，首字母大写 | `Client`, `Matter`, `Payment` |
| DTO | 实体名+DTO | `ClientDTO`, `MatterDTO` |
| Command | 动词+名词+Command | `CreateClientCommand` |
| Repository | 实体名+Repository | `ClientRepository` |
| AppService | 实体名+AppService | `ClientAppService` |
| Controller | 实体名+Controller | `ClientController` |
| Mapper | 实体名+Mapper | `ClientMapper` |

### 2.3 依赖规则

```
interfaces (接口层)
    ↓ 依赖
application (应用层)
    ↓ 依赖
domain (领域层)
    ↓ 依赖
infrastructure (基础设施层)
    ↓ 依赖
common (通用层)

⚠️ 禁止反向依赖！
⚠️ 同层模块之间尽量减少依赖！
```

## 四、开发步骤模板

开发一个新模块时，按以下步骤进行：

### Step 1: 创建实体（domain/xxx/entity）

```java
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
@EqualsAndHashCode(callSuper = true)
@TableName("表名")
public class XxxEntity extends BaseEntity {
    // 字段定义
}
```

### Step 2: 创建Mapper（infrastructure/persistence/mapper）

```java
@Mapper
public interface XxxMapper extends BaseMapper<XxxEntity> {
    // 自定义查询方法
}
```

### Step 3: 创建Repository（domain/xxx/repository）

```java
@Repository
public class XxxRepository extends BaseRepository<XxxMapper, XxxEntity> {
    // 仓储方法
}
```

### Step 4: 创建DTO和Command（application/xxx/dto、command）

### Step 5: 创建AppService（application/xxx/service）

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class XxxAppService {
    private final XxxRepository repository;
    // 业务方法
}
```

### Step 6: 创建Controller（interfaces/rest/xxx）

```java
@RestController
@RequestMapping("/xxx")
@RequiredArgsConstructor
public class XxxController {
    private final XxxAppService appService;
    // API接口
}
```

## 六、Git 工作流

```bash
# 1. 从develop分支创建功能分支
git checkout develop
git pull origin develop
git checkout -b feature/crm-client

# 2. 开发并提交
git add .
git commit -m "feat(crm): 添加客户管理功能"

# 3. 推送并创建PR
git push origin feature/crm-client
# 在GitHub/GitLab创建Pull Request

# 4. Code Review后合并到develop
```

## 七、注意事项

1. **不要修改common和infrastructure层**的已有代码，如需扩展请先讨论
2. **每个模块独立开发**，减少跨模块依赖
3. **写好注释**，特别是业务逻辑
4. **编写单元测试**，覆盖核心逻辑
5. **遵循命名规范**，保持代码风格一致
