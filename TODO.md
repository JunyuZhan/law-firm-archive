# 档案管理系统开发计划

> 基于国家档案局标准（DA/T 104-2024）和国际最佳实践，以**电子档案管理**为核心构建现代化档案管理系统。

## 系统定位

**定位**：接收律所管理系统归档的电子档案，提供专业的电子档案全生命周期管理服务。

**核心理念**：
- 电子档案优先（E-First）
- 全生命周期管理（Lifecycle Management）
- 智能化处理（AI-Powered）
- 合规性保障（Compliance）

## 律所业务契合度核查台账（2026-03-30）

### 核查基准流程

- 纸质档案整理
- 扫描形成电子文件
- 电子档案入库登记
- 电子档案保存与检索利用
- 借阅申请、审批、外链查阅与归还

### 已基本契合的点

| 编号 | 业务要求 | 当前情况 | 结论 |
|------|----------|----------|------|
| FIT-01 | 电子档案为主，纸质位置为辅 | 系统支持 `ELECTRONIC/PHYSICAL/HYBRID` 三种档案形式，纸质/混合档案可记录存放位置、盒号 | ✅ 契合 |
| FIT-02 | 扫描后作为电子档案入库 | 新建档案页采用“上传文件 → 填写信息 → 确认提交”三步式流程，可先上传文件再建档 | ✅ 契合 |
| FIT-03 | 纸质扫描整理要有留痕 | 补充上传已支持卷号、分段、文件来源、扫描批次、扫描人、扫描时间、复核状态、复核人 | ✅ 契合 |
| FIT-04 | 保存后便于查找 | 已有档案列表检索和全文检索，支持关键词、高级筛选、正文检索、高亮 | ✅ 契合 |
| FIT-05 | 借阅方便且可控 | 已支持借阅申请、审批、借出、归还、续借、逾期提醒、外链借阅、下载权限控制 | ✅ 契合 |
| FIT-06 | 借阅类型要和载体对应 | 在线查阅/下载依赖电子载体，复制利用依赖纸质载体，且密级限制下载 | ✅ 契合 |
| FIT-07 | 档案组织应考虑分类体系 | 后端支持全宗、分类、保管期限、状态等结构化字段，分类管理页已存在 | ✅ 基本契合 |

### 当前不契合或存在断点的点

| 编号 | 问题 | 影响 | 优先级 | 建议 |
|------|------|------|--------|------|
| GAP-01 | 手工建档后缺少明确“正式归档”前端入口 | 档案创建成功后未必进入 `STORED`，会直接导致“不能借阅” | P0 | 在档案详情页或列表页提供“正式归档”按钮，并增加状态说明 |
| GAP-02 | 借阅依赖“已归档”，但用户侧提示不够完整 | 律师/档案员可能不知道为什么上传完文件仍无法借阅 | P0 | 在新建成功页、详情页、列表页增加“当前未正式归档”的提示 |
| GAP-03 | 手工入库页未暴露全宗、分类字段 | 后端虽支持标准化组织，但前端手工建档无法完整落实“分库、分类” | P1 | 在新建档案页补充全宗、分类选择，必要时按分类自动带出默认期限 |
| GAP-04 | “分年”更多体现在日期筛选，不是明确的归档组织动作 | 对档案员而言，年度组织感知不强 | P2 | 在列表/详情中增强归档年度展示，必要时增加年度维度入口 |
| GAP-05 | 当前系统配置中的“保管期限”主要是查看，不是完整字典管理 | 未来若律所调整保管期限口径，前后端可能不一致 | P2 | 后续再做正式 CRUD，不作为当前首要问题 |

### 14.1 数据库设计

| 任务ID | 任务名称 | 说明 | 工时 | 状态 |
|--------|----------|------|------|------|
| BE-10.1 | **电子借阅链接表** | 创建 arc_borrow_link 表，存储借阅链接信息 | 0.5天 | ✅ |

**表结构设计**：
```sql
CREATE TABLE arc_borrow_link (
    id BIGSERIAL PRIMARY KEY,
    borrow_id BIGINT,                            -- 关联的借阅申请（可为空，外部系统直接申请时无需关联）
    archive_id BIGINT NOT NULL,                  -- 档案ID
    archive_no VARCHAR(100),                     -- 档案号（冗余）
    access_token VARCHAR(100) NOT NULL UNIQUE,   -- 访问令牌（UUID）
    
    -- 申请信息
    source_type VARCHAR(50),                     -- 来源类型：INTERNAL/LAW_FIRM
    source_system VARCHAR(100),                  -- 来源系统名称
    source_user_id VARCHAR(100),                 -- 来源系统用户ID
    source_user_name VARCHAR(100),               -- 来源系统用户姓名
    borrow_purpose TEXT,                         -- 借阅目的
    
    -- 链接配置
    expire_at TIMESTAMP NOT NULL,                -- 过期时间
    max_access_count INT,                        -- 最大访问次数（NULL不限制）
    allow_download BOOLEAN DEFAULT true,         -- 是否允许下载
    
    -- 访问统计
    access_count INT DEFAULT 0,                  -- 访问次数
    download_count INT DEFAULT 0,                -- 下载次数
    last_access_at TIMESTAMP,                    -- 最后访问时间
    last_access_ip VARCHAR(50),                  -- 最后访问IP
    
    -- 状态
    status VARCHAR(20) DEFAULT 'ACTIVE',         -- ACTIVE/EXPIRED/REVOKED
    revoke_reason TEXT,                          -- 撤销原因
    revoked_at TIMESTAMP,                        -- 撤销时间
    revoked_by BIGINT,                           -- 撤销人
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by BIGINT
);

CREATE INDEX idx_borrow_link_token ON arc_borrow_link(access_token);
CREATE INDEX idx_borrow_link_archive ON arc_borrow_link(archive_id);
CREATE INDEX idx_borrow_link_status ON arc_borrow_link(status);
CREATE INDEX idx_borrow_link_expire ON arc_borrow_link(expire_at);
```

- `ArchiveIndexServiceImpl.addFilters(...)` 当前按请求参数过滤，不按当前用户身份追加范围过滤
- `FileStorageServiceImpl.assertFileAccessAuthorized(...)` 当前主要逻辑为：
  - 管理员/档案员直接放行
  - 普通用户依赖借阅申请判断
  说明：文件访问控制有一定基础，但不等于档案数据权限体系完整
